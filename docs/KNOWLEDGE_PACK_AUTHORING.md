# Knowledge Pack Authoring Guide

## 中文

### 1. 知识包目标

ReiLink 的知识包不是完整攻略站，也不是爬取 wiki 或攻略网站后的数据库。知识包只用于给 Rei 提供少量、可控、简短的事实 context，例如 boss 名称、位置、机制提示或新手容易卡住的玩法信息。

最终回复仍由 LLM + persona 生成。知识包只提供事实材料，不直接生成 Rei 的话，也不应该让产品变成攻略站。

### 2. 目录结构

```text
data/knowledge/games/catalog.json
data/knowledge/games/{game_id}/manifest.json
data/knowledge/games/{game_id}/snippets.json
```

`catalog.json` 管理所有游戏条目。每个 supported 游戏应有自己的目录，并包含 `manifest.json` 与 `snippets.json`。

### 3. 新增游戏流程

1. 在 `catalog.json` 添加 game entry。
2. 新建 `data/knowledge/games/{game_id}/`。
3. 添加 `manifest.json`。
4. 添加 `snippets.json`。
5. 运行 `make validate-knowledge`。
6. 测试 Debug Panel / Prompt Preview 是否匹配当前游戏、主题和 snippet。
7. 确认可用后，从 `planned` 升级为 `supported`。

### 4. catalog.json 示例

```json
{
  "game_id": "hollow_knight",
  "display_name": "空洞骑士",
  "aliases": ["Hollow Knight", "空洞骑士", "空洞"],
  "manifest_path": "data/knowledge/games/hollow_knight/manifest.json",
  "knowledge_path": "data/knowledge/games/hollow_knight/snippets.json",
  "enabled": true,
  "knowledge_available": true,
  "support_status": "supported"
}
```

字段说明：

- `game_id`：稳定的游戏 ID，使用小写 snake_case。
- `display_name`：界面和调试信息中显示的游戏名，优先使用简体中文。
- `aliases`：用户可能输入的中英文别名、简称或常见称呼。
- `manifest_path`：该游戏 `manifest.json` 的仓库相对路径。
- `knowledge_path`：该游戏 `snippets.json` 的仓库相对路径。
- `enabled`：是否启用该 catalog entry。
- `knowledge_available`：是否已有可用知识包。
- `support_status`：支持状态，见“支持状态说明”。

### 5. manifest.json 示例

```json
{
  "game_id": "hollow_knight",
  "display_name": "空洞骑士",
  "version": "0.1.0",
  "language": "zh-CN",
  "status": "sample",
  "knowledge_files": ["snippets.json"],
  "coverage": ["boss", "mechanic", "beginner_tip"],
  "last_updated": "2026-05-31",
  "notes": "Minimal sample knowledge pack for validating multi-game retrieval."
}
```

`manifest.json` 描述知识包本身，而不是游戏的完整资料。`coverage` 应保持简短，用于说明当前 sample pack 覆盖了哪些主题。

### 6. snippets.json 示例

```json
[
  {
    "id": "mantis_lords_rhythm",
    "title": "螳螂领主：节奏观察",
    "type": "boss_strategy",
    "text": "螳螂领主的威胁主要来自连续冲刺和下劈。先站在中间看落点，少追击，等短停顿再补一刀。",
    "aliases": ["螳螂领主", "Mantis Lords", "螳螂"],
    "tags": ["boss", "rhythm", "positioning"]
  },
  {
    "id": "early_exploration_map",
    "title": "新手探索：先找地图",
    "type": "beginner_tip",
    "text": "进入新区域后优先听纸张声找制图师。买地图后再配合羽毛笔补全路线，迷路时先回长椅整理方向。",
    "aliases": ["地图", "制图师", "Cornifer", "探索"],
    "tags": ["exploration", "map", "beginner_tip"]
  }
]
```

每条 snippet 应至少包含 `id`、`title`、`type`、`text`。当前校验工具也兼容现有 sample packs 中的 `kind` / `summary` 字段，但新增知识包建议优先使用 `type` / `text`。

### 7. Snippet 写作规范

- 内容要短，适合被 Rei 转述成 1-3 句。
- 使用原创概括，不复制外部攻略原文。
- 每条 snippet 聚焦一个事实或技巧。
- 不写长篇攻略。
- 不写大量剧情设定。
- 不直接模仿攻略站的结构和语气。
- 尽量包含 `aliases`，方便检索用户的自然说法。
- `tags` 应该简短、可检索，避免太长或太泛。

### 8. 支持状态说明

- `supported`：已支持，有可用知识包，必须配置有效的 `manifest_path`、`knowledge_path` 和 `knowledge_available: true`。
- `planned`：计划支持，但还没有可用知识包。
- `detected_only`：可识别游戏，但没有知识库。
- `unsupported`：暂不支持。

### 9. 校验知识包

```bash
make validate-knowledge
```

该命令会检查：

- `catalog.json` 结构。
- supported 游戏的 `manifest.json`。
- supported 游戏的 `snippets.json`。
- duplicate ids。
- missing text。
- missing paths。
- supported 游戏是否有知识包。

如果只有 warning，例如 planned 游戏暂未接入知识包，命令仍会以 exit code 0 结束。存在 error 时 exit code 为 1。

### 10. 调试知识匹配

可以通过 UI Debug Panel / Prompt Preview 检查知识匹配状态：

- 当前游戏。
- 知识库状态。
- 匹配主题。
- snippet 标题。
- `knowledge_used_in_prompt`。

这些调试信息用于确认知识包是否被正确识别和注入 prompt context。不要通过硬编码 Rei 回复来验证知识内容。

### 11. 版权和来源注意

- 不要复制粘贴攻略站 / wiki 长段文本。
- 使用自己的简短总结。
- 只记录通用事实和玩法提示。
- 不收集用户隐私。
- 如果未来引入外部来源，需要记录 source metadata。

知识包应保持轻量、可审查、可维护。新增内容时，优先写少量高质量 snippets，而不是堆大量不确定文本。

## English

### 1. Knowledge Pack Goals

ReiLink knowledge packs are not complete guide databases, and they are not scraped wiki or guide-site mirrors. A pack gives Rei a small, controlled, concise set of factual context, such as boss names, locations, mechanics, or beginner tips that players commonly ask about.

Final replies are still generated by the LLM + persona. Knowledge packs provide source material only; they should not directly generate Rei's wording or turn the product into a guide site.

### 2. Directory Structure

```text
data/knowledge/games/catalog.json
data/knowledge/games/{game_id}/manifest.json
data/knowledge/games/{game_id}/snippets.json
```

`catalog.json` lists all game entries. Each supported game should have its own directory with `manifest.json` and `snippets.json`.

### 3. Adding A New Game

1. Add a game entry to `catalog.json`.
2. Create `data/knowledge/games/{game_id}/`.
3. Add `manifest.json`.
4. Add `snippets.json`.
5. Run `make validate-knowledge`.
6. Test the Debug Panel / Prompt Preview for game, topic, and snippet matching.
7. After confirming the pack works, upgrade the game from `planned` to `supported`.

### 4. catalog.json Example

```json
{
  "game_id": "hollow_knight",
  "display_name": "空洞骑士",
  "aliases": ["Hollow Knight", "空洞骑士", "空洞"],
  "manifest_path": "data/knowledge/games/hollow_knight/manifest.json",
  "knowledge_path": "data/knowledge/games/hollow_knight/snippets.json",
  "enabled": true,
  "knowledge_available": true,
  "support_status": "supported"
}
```

Field notes:

- `game_id`: stable game identifier, preferably lowercase snake_case.
- `display_name`: display name used in UI and debug output, Chinese-first when possible.
- `aliases`: Chinese and English names, abbreviations, and common player terms.
- `manifest_path`: repository-relative path to the game's `manifest.json`.
- `knowledge_path`: repository-relative path to the game's `snippets.json`.
- `enabled`: whether the catalog entry is enabled.
- `knowledge_available`: whether a usable knowledge pack exists.
- `support_status`: support state, described below.

### 5. manifest.json Example

```json
{
  "game_id": "hollow_knight",
  "display_name": "空洞骑士",
  "version": "0.1.0",
  "language": "zh-CN",
  "status": "sample",
  "knowledge_files": ["snippets.json"],
  "coverage": ["boss", "mechanic", "beginner_tip"],
  "last_updated": "2026-05-31",
  "notes": "Minimal sample knowledge pack for validating multi-game retrieval."
}
```

`manifest.json` describes the pack, not the entire game. Keep `coverage` short and use it to show what the current sample pack actually covers.

### 6. snippets.json Example

```json
[
  {
    "id": "mantis_lords_rhythm",
    "title": "螳螂领主：节奏观察",
    "type": "boss_strategy",
    "text": "螳螂领主的威胁主要来自连续冲刺和下劈。先站在中间看落点，少追击，等短停顿再补一刀。",
    "aliases": ["螳螂领主", "Mantis Lords", "螳螂"],
    "tags": ["boss", "rhythm", "positioning"]
  },
  {
    "id": "early_exploration_map",
    "title": "新手探索：先找地图",
    "type": "beginner_tip",
    "text": "进入新区域后优先听纸张声找制图师。买地图后再配合羽毛笔补全路线，迷路时先回长椅整理方向。",
    "aliases": ["地图", "制图师", "Cornifer", "探索"],
    "tags": ["exploration", "map", "beginner_tip"]
  }
]
```

Each snippet should include at least `id`, `title`, `type`, and `text`. The validator also accepts the existing sample-pack shape using `kind` / `summary`, but new packs should prefer `type` / `text`.

### 7. Snippet Writing Rules

- Keep content short enough for Rei to paraphrase in 1-3 sentences.
- Write original summaries; do not copy external guide text.
- Focus each snippet on one fact or practical tip.
- Avoid long-form walkthroughs.
- Avoid large amounts of lore or plot summary.
- Do not imitate guide-site structure or tone directly.
- Include `aliases` where possible so natural player phrasing can match.
- Keep `tags` short and searchable.

### 8. Support States

- `supported`: the game has a usable knowledge pack and must provide valid `manifest_path`, `knowledge_path`, and `knowledge_available: true`.
- `planned`: support is planned, but no usable knowledge pack exists yet.
- `detected_only`: the game can be recognized, but has no knowledge pack.
- `unsupported`: not currently supported.

### 9. Validating Knowledge Packs

```bash
make validate-knowledge
```

The command checks:

- `catalog.json` structure.
- `manifest.json` for supported games.
- `snippets.json` for supported games.
- duplicate ids.
- missing text.
- missing paths.
- whether supported games have usable knowledge packs.

Warnings, such as a planned game without a pack, still exit with code 0. Errors exit with code 1.

### 10. Debugging Knowledge Matching

Use the UI Debug Panel / Prompt Preview to inspect:

- current game.
- knowledge status.
- matched topics.
- snippet titles.
- `knowledge_used_in_prompt`.

These fields help confirm that a pack is recognized and injected into prompt context. Do not verify knowledge behavior by hardcoding Rei replies.

### 11. Copyright And Source Notes

- Do not copy long passages from guide sites or wikis.
- Use your own concise summaries.
- Record only general facts and gameplay tips.
- Do not collect user privacy data.
- If external sources are added in the future, record source metadata.

Knowledge packs should remain lightweight, reviewable, and maintainable. Prefer a small set of high-quality snippets over a large pile of uncertain text.
